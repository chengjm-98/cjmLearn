前端选型我一般会先看业务形态和团队情况，再结合性能、可维护性、工程化和扩展性来决定，不会单纯因为框架流行就选。

# 单体前端

适用场景：

- 业务相对集中
- 团队规模不大
- 页面间强关联
  如果业务是一个整体系统，我会采用单体前端架构，通过模块化、路由拆分和组件化来控制复杂度。
  ## 什么时候选择react
  - 我更倾向在复杂业务、状态多、交互重的项目中使用 React。
  - 理由 + 原理：
    - 函数式思想，UI = f(state)
    - 单向数据流，更容易定位问题
    - Hooks 让逻辑复用更灵活
    - 原理点一句就够：
    - React 通过 Virtual DOM 和 Diff 算法减少不必要的 DOM 更新。
  ## 什么时候选择vue
  - 如果项目偏向中小型、开发节奏快，或者团队中前端经验参差不齐，我会更倾向 Vue。
  - 理由 + 原理：
    - 上手成本低
    - 模板语法直观
    - 双向绑定降低心智负担
    - 原理点一句：
    - Vue 基于响应式系统，通过依赖收集和派发更新实现数据驱动视图。

# 微前端

- 多业务线
- 多团队并行开发
- 历史系统多（React + Vue 混用）
- 需要独立部署
  当系统规模扩大、多个团队并行开发，或者需要整合不同技术栈的子系统时，我会考虑微前端方案，比如 qiankun。

## 选择qiankun的原因

- 对主流框架（React / Vue）兼容性好
- 提供完整的应用生命周期管理
- 子应用可以独立开发、构建、部署
- 社区成熟，基于 single-spa 封装
