# git rebase

git rebase 是 Git 中用于 重新应用提交（commits）到新的基础（base）上的操作。它的核心目的是 线性化历史，避免复杂的分叉，让提交历史更清晰。

## 步骤

- git checkout feature git rebase main

## 解决冲突

- git add <file> 标记解决完成。
- git rebase --continue 继续。

## 为什么 git rebase 冲突会多

- rebase 是逐个提交应用，而 merge 是一次性合并。
- 每个提交都可能触发冲突。

## 什么时候不能用 git rebase

- 公共分支（例如 main、master、release）不应该 rebase，因为会改写历史，影响其他人。

## 什么时候适合用 rebase？

- 在自己的 feature 分支上更新主分支代码。
- 在提交到远程之前整理历史，压缩无用 commit。
- 为了线性历史便于 code review。

# git rebase 和 git merge 的区别

| 特性     | `git merge`                            | `git rebase`                     |
| -------- | -------------------------------------- | -------------------------------- |
| 历史结构 | 保留分支历史，可能形成“合并节点”       | 重新写历史，使提交历史线性化     |
| 操作结果 | 会产生一个新的合并提交（merge commit） | 不产生合并提交（除非冲突处理时） |
| 易读性   | 分支图可能复杂                         | 历史更干净、线性                 |
